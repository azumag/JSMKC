# TA Phase 3: Top 2 の個別ライフ調整仕様（CDM向け）

## 1. 目的

通常TAのPhase 3は、開始時および残り8人・4人・2人に到達した際に、残存選手のライフを3へ戻す。
CDMのTop 2では、運営判断により、この自動リセット後に各選手を**5ライフ**で進行させる場合がある。

大会全体の通常ルール（3ライフ）を変更せず、管理者が決勝画面で各アクティブ選手の現在ライフを明示的に設定できるようにする。これはCDM直前の運営復旧機能であり、Top 2の自動ルール変更ではない。

## 2. 適用範囲

- 対象: 通常TAのPhase 3（`/tournaments/[id]/ta/finals`）
- 対象ユーザー: 管理者のみ
- 対象選手: 対象大会に属する、未敗退のPhase 3エントリ
- 値域: 1〜10の整数
- 想定運用: Top 2到達後、必要な大会では次ラウンド開始前に、残っている2名をそれぞれ5へ設定する

通常TAの初期3ライフ、8/4/2人時の3ライフ自動リセット、TAバトルロワイヤルの10ライフ・回復なしのルールは変更しない。

## 3. 管理画面の仕様

決勝順位表の各アクティブ選手行に、管理者だけが操作できる以下を表示する。

- 現在ライフを初期値とする数値入力（1〜10）
- 「ライフを保存」ボタン

Top 2到達時にも固定の警告は表示しない。ライフ調整は運営者が必要な大会で任意に行う操作であり、自動補正も行わない。

保存時は、入力した目標値をそのまま設定する。`+2`のような差分操作ではないため、3から5への変更を明確に行える。

次の場合は入力・保存を無効にする。

- 未提出のラウンドが存在する
- ラウンド送信中または開始処理中
- 未解決のサドンデスが存在する
- 大会がアーカイブ済み
- 対象ステージが凍結済み

敗退済み選手にはライフ調整操作を出さない。ライフを0にする、または敗退者を復帰させる操作はこの機能の対象外であり、既存の敗退・ラウンド訂正の正規手順を使う。

ライフ表示は、手動設定値が既定の3を超える場合でも誤解を生まないよう、決勝では `♥ 5` のように**現在値**を表示する。`♥ 5/3` のような既定値との比率表示はしない。

## 4. API契約

既存の差分更新`update_lives`はE2Eなどの既存利用との互換性のため残す。新しい管理画面は、目標値を設定する専用アクションだけを使う。

```http
PUT /api/tournaments/{tournamentId}/ta
Content-Type: application/json

{
  "entryId": "<TTEntry CUID>",
  "action": "set_lives",
  "lives": 5,
  "expectedVersion": 7,
  "expectedLives": 3
}
```

成功時は更新後のエントリ（`lives: 5`, `version: 8`）を返す。

### 4.1 サーバー側の不変条件

- 管理者以外は403。
- `entryId`はURLの大会に所属していなければ404。
- `stage === "phase3"`かつ`eliminated === false`以外は409。
- `lives`は1〜10の整数以外を400。
- 凍結ステージは既存の凍結エラー（403）を返す。
- `expectedVersion`または`expectedLives`が現在の値と異なる場合は409 `OPTIMISTIC_LOCK_ERROR`。
- 成功時に`lives`と`version`を更新し、同じD1 atomic batchで絶対値の`TTPhaseLifeAdjustment`イベントを保存する。イベントには調整前後の値、実行日時、実行者、直前の確定ラウンドを残す。
- 監査ログには大会ID、Phase 3、選手、変更前ライフ、変更後ライフ、`manualUpdate: true`を残す。

この楽観ロックにより、管理者が画面を開いている間にラウンド結果が確定しても、古い画面から「5」を保存して新しい結果を上書きすることはない。管理者は再読み込み後に、必要なら改めて設定する。

## 5. 運営手順（Top 2で調整が必要な場合）

1. Phase 3で残り2名に到達するまで、通常ルールどおり進行する。
2. ラウンドの送信と、必要なサドンデスの解決が完了したことを確認する。
3. 決勝順位表で残存する各選手のライフを5に設定して保存する。
4. 表示が両者とも`♥ 5`になったことを確認する。
5. 次ラウンドを開始する。下位側は通常どおりライフを1失い、5から4になる。

## 6. 再生・履歴

- Phase 3の現在ライフは、確定ラウンドと`TTPhaseLifeAdjustment`を実行時刻順に再生して復元する。
- 手動調整は差分ではなく「指定値への設定」なので、再計算を再試行しても二重加算されない。
- ライフリセットは同じ時系列イベントとして後続するため、リセットより前の調整はリセット後に再適用されない。
- `undo` / `cancel_last_round` は対象ラウンドだけを再生列から除き、手動調整は削除しない。調整後にラウンドを再送信した場合も、保存時刻に従って新しいラウンドとの前後関係を決める。
- `cancel_last_round` は対象ラウンドを先に未提出状態へフェンスする。並行するライフ調整は、フェンス前に完了して再生対象になるか、フェンス後に409で拒否されるかのどちらかになり、成功イベントを復元処理で上書きしない。
- ラウンド履歴には調整前後の値、実行日時、実行者をラウンドと同じ時系列で表示する。
- PR #3042 で既に作られた調整は構造化監査ログからマイグレーション時にバックフィルし、保存済み大会でも再生可能にする。
- **アーカイブ再生とCDM Excel出力**: 現行のアーカイブ形式とCDM出力は手動調整イベントを含めない。ライブ大会の整合性修復とは別に、アーカイブ形式のバージョン更新が必要になる。
- **Top 2の自動5ライフ化**: 今回は行わない。将来これを正式ルールにする場合は、閾値ごとの回復ライフ値をデータモデル化し、ラウンド履歴・Undo・アーカイブ・CDM出力まで一貫して変更する。

## 7. 受入基準

1. 管理者は通常TA Phase 3のアクティブ選手を3から5へ設定できる。
2. 非管理者、敗退選手、他大会のエントリでは設定できない。
3. 競合した保存は409になり、ライフ値を変更しない。
4. 未提出ラウンド・送信中・未解決サドンデス・アーカイブ中は画面操作が無効である。
5. Top 2到達時にも固定の5ライフ警告は表示されない。管理者は必要な場合にだけ任意の値へ変更でき、自動的には変更されない。
6. 5へ設定した選手が次の通常ラウンドでライフを失うと、4として進行する。
7. 監査ログに変更前後の値が残る。
8. E2Eでは、Phase 3の2名を管理画面から5へ設定し、Phase APIで両者の現在ライフが5であることを確認する。
9. 手動調整の前後にあるラウンドをundo/cancelしても、有効な調整を一度だけ再生して現在ライフと活動状態を維持する。
10. ラウンド履歴で調整前後の値、実行日時、実行者を確認できる。
11. cancelと並行した調整は、成功して再生に含まれるか409で拒否され、イベントと現在ライフを不一致にしない。
