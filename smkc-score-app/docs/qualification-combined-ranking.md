# 2Pモード（BM/MR/GP）予選 — 合算順位・決勝進出ライン 仕様書

対象: グループ予選が3グループ以上に分かれた場合の「合算順位」（全グループを横断した1本の順位）の決定ルール、および決勝進出ライン（直接勝ち上がり／バラージ／敗退）の3グループ以上への一般化。

**本書は仕様の整理のみを目的とし、コード変更は含まない。** 現状コードは3グループ以上のグループ分けをUI（`group-setup-dialog.tsx`）・API（`qualification-route.ts`）の両方で明示的にロックしており、決勝進出ロジック（`finals-group-selection.ts`）も3グループ以上は「ラウンドロビン式の仮並べ」のままで、コード内コメントに「combined-ranking rules for 3+ groups are a separate follow-up」と明記されている。本書はその保留部分を定義する。

## 1. 前提（現状すでに正しく実装されている部分）

グループ**内**の順位は `src/lib/server-ranking.ts` の `assignRanksForPartition()` によってすでに正しく計算されている。合算順位はこの「グループ内最終順位」を入力として使うため、以下は変更しない。

グループ内タイブレークの順序（既存実装）:

1. **WDLスコア（勝点 = 勝×2＋分×1）** 降順
2. **得失点差（BM/MR）／ドライバーズポイント累計（GP）** 降順
   - BM/MR: `winRounds - lossRounds`（`bm-config.ts:123`, `mr-config.ts:135`）
   - GP: レースごとの獲得ポイント累計、差分ではない（`gp-config.ts:7-8, 188-189`）
3. **直接対決（H2H）**: 上記までで同点のグループ内選手同士の対戦結果のみで再ソート（`server-ranking.ts:59-137`）
4. **管理者によるサドンデス結果の手動確定**（`rankOverride`。UI上は「サドンデスプレーオフ」。`qualification-playoff-manager.tsx`）

「シードランキング」（`seeding` フィールド）は現状、グループ分け・組み合わせ表（総当たり順）決定にのみ使われ、順位のタイブレークには使われていない。本書で新たにタイブレークとしての役割を追加する。

## 2. 合算順位（グループ横断）決定ルール

### 2.1 優先順位

グループ内最終順位（1.の結果）を入力として、以下の優先順位で全グループの選手を1本の順位に並べる。

1. **グループ内順位** — 異なれば、他の指標に関わらず無条件でグループ内順位が良い方が上位
2. **WDLスコア（勝点）**
3. **シードランキング**（予選開始前に決定した値。数値が小さい方が上位）
4. **得失点差（BM/MR）／ドライバーズポイント（GP）**
5. **サドンデス**（新たな対戦）

### 2.2 「グループ内順位バケット」という考え方

全グループの「K位」を1つの束（バケット）として扱う。バケット同士は絶対順位であり、**バケットKの全員は、バケットK+1の全員より必ず上位になる**（バケット内の個々の成績が劣っていても関係ない）。2.1の②〜⑤は同一バケット内のタイブレークとしてのみ働く。

例: A6（グループA内6位）は勝点・得失点差ともB5（グループB内5位）を上回っていても、バケット6はバケット5より下なので、B5がA6より上位になる。

### 2.3 グループの人数が不揃いな場合

3グループ以上では人数が均等に割り切れないことがある（例: 32人→3グループ＝11/11/10人）。

バケットKを作る際、**その時点で実在する選手を持つグループだけをバケットに含める**。人数が足りないグループは、そのバケット以降は単に存在しないだけとして扱う。追加のエラー処理や自動最下位化は不要。

これは `round-robin.ts` の `__BREAK__`（グループ内が奇数人のとき、対戦表を偶数に揃えるための仮想対戦相手。実プレイヤーは自動勝利となり順位表には出てこない）と同じ考え方 ——「存在しない枠は静かに欠番となり、実在する選手同士の比較を歪めない」—— の転用である。ただし `__BREAK__` 自体（`BREAK_PLAYER_ID`）は総当たり組み合わせ生成専用のスケジューリング上の仕組みであり、合算順位のバケット構築のために `__BREAK__` の実レコードを新設する必要はない。

### 2.4 サドンデスの運用

- **グループ内タイ**（1.の④）: 既存の運用を継続。管理者が同グループの直接対戦結果、または新たに実施した決着試合の結果を見て `rankOverride` で手動確定する。
- **グループ横断タイ**（2.1の⑤、上記①〜④すべてが同点の場合）: 対象選手は一度も対戦していないため、既存の対戦結果は使えない。**新たに決着試合を組んで対戦し**、結果を既存の `rankOverride` の仕組みで記録する。3人以上のタイの場合は決着試合の総当たり、または既存の「サドンデスプレーオフ」UIと同様の運用（管理者が結果を見て順位を確定）を想定する。

## 4. 決勝進出ライン（3グループ以上への一般化）

### 4.1 現行（2グループ）の仕組み

`src/lib/finals-group-selection.ts` を参照。決勝進出枠の総数は固定値 `TOTAL_FINALS_SLOTS = 12`（参加人数から算出されるものではない）。

```
perGroup = 12 / グループ数
各グループの 1〜perGroup 位     → 直接勝ち上がり（Upper Bracket）
各グループの (perGroup+1)〜(2*perGroup) 位 → バラージ（サバイバルトーナメント）
各グループの (2*perGroup+1) 位以降        → 敗退
```

2グループの場合 `perGroup = 6`。直接12人＋バラージ12人（4人がバラージを勝ち上がりTop16入り）＝ Top16。これはPDFの「Top16確定ライン」（合算順位1〜12位）「予選通過ライン」（13〜24位がサバイバルトーナメントへ）と一致する。

重要な点: この「グループごとの固定人数」方式は、**グループの人数が揃っている場合に限り**、2章の合算順位で1〜12位を取った場合と数学的に一致する（「グループ内順位が異なれば絶対優先」というルールにより、バケットKには各グループから必ず1人ずつしか入らないため）。

### 4.2 3グループへの一般化

```
perGroup = 12 / 3 = 4
各グループ 1〜4位   → 直接勝ち上がり（計12人）
各グループ 5〜8位   → バラージ（計12人）
各グループ 9位以降  → 敗退
```

決勝進出枠の総数（Top16 = 直接12＋バラージ12）はグループ数によらず固定とする。

グループ人数が不揃い（例: 11/11/10人）でも、各グループが最低 `2 * perGroup = 8` 人いれば成立する（既存コードの `selectFinalsEntrantsByGroup` が持つバリデーションと同じ条件）。

### 4.3 スコープ外

決勝ブラケット内の**シード番号配置**（同グループ対決を避けるための並び順。2グループでは `TWO_GROUP_DIRECT_UPPER_SEEDS` / `TWO_GROUP_BARRAGE_SEED_TOKENS` という固定トークンマップで定義されている）は、本書の対象外とする。既存コードのコメント通り、別タスクとして切り出す。3グループの直接勝ち上がり・バラージの**対象者選定**（4.2）は本書で定義するが、**その並び順**は今後別途検討する。

## 5. 用語対応表

| 本書での呼称 | コード上の名称 | 備考 |
|---|---|---|
| グループ内順位 | `assignRanksForPartition()` の出力（`_rank`） | すでに実装済み、変更なし |
| WDLスコア／勝点 | `score`（`wins*2+ties`） | 全モード共通の計算式 |
| 得失点差 | `points`（BM/MR） | `winRounds - lossRounds` |
| ドライバーズポイント | `points`（GP） | レース獲得点の累計（差分ではない） |
| シードランキング | `seeding` | 現状は組分け・組み合わせ表決定のみに使用。本書でタイブレークとしての用途を追加 |
| サドンデス | `rankOverride` / `rankOverrideBy` / `rankOverrideAt`（UI上「サドンデスプレーオフ」） | グループ内タイは既存運用のまま。グループ横断タイは新規対戦が必要 |
