# 2Pモード（BM/MR/GP）予選 — 合算順位・決勝進出ライン 仕様書

対象: グループ予選が3グループ以上に分かれた場合の「合算順位」（全グループを横断した1本の順位）の決定ルール、および決勝進出ライン（直接勝ち上がり／バラージ／敗退）の3グループ以上への一般化。

**本書は仕様の整理のみを目的とし、コード変更は含まない。** 現状コードは3グループ以上のグループ分けをUI（`group-setup-dialog.tsx`）・API（`qualification-route.ts`）の両方で明示的にロックしており、決勝進出ロジック（`finals-group-selection.ts`）も3グループ以上は「ラウンドロビン式の仮並べ」のままで、コード内コメントに「combined-ranking rules for 3+ groups are a separate follow-up」と明記されている。本書はその保留部分を定義する。

## 0. 更新履歴・本書の位置付け

初版（2.1〜2.4, 3章）は「WDLスコア→シードランキング→得失点差→サドンデス」という**あるべきタイブレーク案**を、添付PDF（実イベントの合算順位結果）の誤りと思われる4件から逆算して提案したものだった。

その後、実際の CDM Excel テンプレート（`public/templates/cdm-2025-template.xlsm`。CDM 2025 実イベントの完成ワークブックと同一内容。以下「CDM Excel」）の数式を精査した結果、**CDM Excel は初版が提案したタイブレークを一切実装していない**ことが判明した。添付PDFの結果は（人手修正等の可能性はあるが）少なくとも「CDM Excelの数式がシードランキングでタイブレークしている」という初版の仮説では説明できない。

本改訂では、2章・3章を **CDM Excelが実際に計算している内容** に合わせて書き換え、初版のタイブレーク案は7章「質問事項」に持ち越して要検討事項として残す。4章（決勝進出ライン）・6章（アプリ実装への影響範囲）はアプリ側 (`finals-group-selection.ts` 等) の将来実装についての整理であり、CDM Excelの現状とは別レイヤーの話のため変更していない。

## 1. 前提（現状すでに正しく実装されている部分）

グループ**内**の順位は `src/lib/server-ranking.ts` の `assignRanksForPartition()` によってすでに正しく計算されている。合算順位はこの「グループ内最終順位」を入力として使うため、以下は変更しない。

グループ内タイブレークの順序（既存実装）:

1. **WDLスコア（勝点 = 勝×2＋分×1）** 降順
2. **得失点差（BM/MR）／ドライバーズポイント累計（GP）** 降順
   - BM/MR: `winRounds - lossRounds`（`src/lib/event-types/bm-config.ts:123`, `src/lib/event-types/mr-config.ts:135`）
   - GP: レースごとの獲得ポイント累計、差分ではない（`src/lib/event-types/gp-config.ts:7-8, 188-189`）
3. **直接対決（H2H）**: 上記までで同点のグループ内選手同士の対戦結果のみで再ソート（`server-ranking.ts:59-137`）
4. **管理者によるサドンデス結果の手動確定**（`rankOverride`。UI上は「サドンデスプレーオフ」。`qualification-playoff-manager.tsx`）

「シードランキング」（`seeding` フィールド）は現状、グループ分け・組み合わせ表（総当たり順）決定にのみ使われ、順位のタイブレークには使われていない。

CDM Excel側もこのグループ内順位に相当する中間列を持つが、H2H・サドンデス（管理者手動確定）は数式では表現不可能なため反映されていない（詳細は2.1参照）。

## 2. 合算順位（グループ横断）決定ルール — CDM Excelの実装に基づく現状仕様

### 2.1 CDM Excelが実際に行っている計算

対象シート: `BM Qualifications` / `MR Qualifications` / `GP Qualifications`（列名はモードごとに異なるが構造は共通。BM: `AF`〜`AL`列、MR: `AJ`〜`AP`列、GP: `AG`〜`AM`列。5章の対応表参照）。

まずグループ内ランキングをExcel数式で再現する:

```
SORTBY(選手名, グループ文字(A/B/C/…) 昇順, Score 降順, Points 降順)
```

- `Score`＝`INT(1000*W/MP + 500*T/MP)`（`W`=勝数, `T`=分数, `MP`=消化試合数。全員同数のため`wins*2+ties`と単調に一致し順位への影響は実質同一）
- `Points`＝`winRounds - lossRounds`（BM/MR）／レース獲得ポイント累計（GP）
- H2H・サドンデスは数式で表現できないため**含まれない**（1章の通りアプリ側のみの機能）

続いて、上記を「グループ内順位バケット」でグループ横断1本の順位に並べ替える最終式が下記（BM `AL`列 / MR `AP`列 / GP `AM`列。GPはこの列がそのまま `GP Finals!B3` から `XLOOKUP` され決勝シードの基礎になっており、表示専用ではなく実際の決勝進出判定に使われる列であることを確認済み）:

```
IF(グループ数 = 3,
   SORTBY(グループ内ランキング, グループ内順位バケット 昇順, Score 降順, Points 降順),
   SORTBY(グループ内ランキング, グループ内順位バケット 昇順)   ← 2グループ運用は常にこちら
)
```

すなわち実装済みの優先順位は:

1. **グループ内順位（バケット）** — 異なれば無条件でグループ内順位が良い方が上位（初版と同じ）
2. **グループ数がちょうど3のときだけ**: WDLスコア（Score）降順 → 得失点差／ポイント（Points）降順
3. **上記2グループ・4グループ以上**（＝現行の2グループ運用を含む）: **追加のタイブレークは一切ない**。同バケット内は`SORTBY`が安定ソートであるため、元の配列順＝**グループ文字のアルファベット順（A→B→C→D）がそのまま結果になる**。成績・シードの比較は行われない。

**シードランキング（`seeding`）は、2グループ・3グループいずれの数式にも一度も登場しない。** 初版2.1が提案した「シードランキングでタイブレーク」は、CDM Excelには実装されていない。

3グループ専用の分岐は特定イベント向けの個別対応と見られ、4グループ以上には一般化されていない（`VSTACK`自体もA〜Dの4グループまでしかハードコードされていない）。

### 2.2 「グループ内順位バケット」という考え方

全グループの「K位」を1つの束（バケット）として扱う。バケット同士は絶対順位であり、**バケットKの全員は、バケットK+1の全員より必ず上位になる**（バケット内の個々の成績が劣っていても関係ない）。この点は初版・CDM Excelどちらも共通。

ただし同一バケット内の並びは、2.1の通り**現行の2グループ運用ではグループ文字順で固定**であり、成績（WDLスコア・得失点差）による入れ替えは起きない。

例: 現行のCDM Excelでは、A6（グループA内6位）がB5（グループB内5位）より勝点・得失点差で優れていても、そもそも異なるバケット（6 vs 5）なのでB5が上位になる（ここまでは初版と同じ）。一方、**同じバケット**内のA4とB4では、B4の方が勝点で上回っていても（3章参照）、常にA4が上位になる。

### 2.3 グループの人数が不揃いな場合

**アプリ側の理想（初版の提案、未実装）**: 3グループ以上では人数が均等に割り切れないことがある（例: 32人→3グループ＝11/11/10人）。バケットKを作る際、その時点で実在する選手を持つグループだけをバケットに含め、人数が足りないグループはそのバケット以降は単に存在しないだけとして扱う、という考え方は引き続き妥当な提案として残す。

**CDM Excelの現状**: `SEQUENCE('Main Hub'!P2/'Main Hub'!P4)` のように総人数を「グループ数」で単純に割った値を行数として使っており、**割り切れない（不均等な）グループ構成を想定していない**（`cdm-export-design.md` 3.1.1も「制約: シート数式は均等グループ（P2/P4が整数）を前提とする」と明記）。不均等な場合にCDM Excel側がどう振る舞うか（エラーになるか、行数が丸められて選手が欠落・重複するか）は未検証。少なくとも「欠けたグループのバケットを静かにスキップする」という初版の設計をExcelの数式が実現しているわけではない。

これは `round-robin.ts` の `__BREAK__`（グループ内が奇数人のとき、対戦表を偶数に揃えるための仮想対戦相手。実プレイヤーは自動勝利となり順位表には出てこない）と同じ考え方 ——「存在しない枠は静かに欠番となり、実在する選手同士の比較を歪めない」—— の転用としてアプリ側で新たに実装する分には引き続き有効な方針だが、CDM Excel側にはこれに相当する仕組みは存在しない。

### 2.4 サドンデスの運用

- **グループ内タイ**: 既存の運用を継続。管理者が同グループの直接対戦結果、または新たに実施した決着試合の結果を見て `rankOverride` で手動確定する。
- **グループ横断タイ**（2.1②〜③がいずれも同点、またはCDM Excelの現状のようにタイブレーク自体が存在しない場合）: 対象選手は一度も対戦していないため、既存の対戦結果は使えない。新たに決着試合を組んで対戦し、結果を既存の `rankOverride` の仕組みで記録する、という初版の提案は変更なし。

CDM Excelには数式でサドンデスを表現する仕組みはなく、これはアプリ側（または大会運営の手作業）でのみ対応可能な事項である。

> **決定（大会運営確認済み）**: 3グループ専用分岐（WDLスコア→得失点差）まで適用してもなお決着しない完全同点は、上記のグループ横断タイと同じくサドンデスプレーオフで決着する。7章の質問1参照。

## 4. 決勝進出ライン（3グループ以上への一般化）

### 4.1 現行（2グループ）の仕組み

`src/lib/finals-group-selection.ts` を参照。決勝進出枠の総数は固定値 `TOTAL_FINALS_SLOTS = 12`（参加人数から算出されるものではない）。

```
perGroup = 12 / グループ数
各グループの 1〜perGroup 位     → 直接勝ち上がり（Upper Bracket）
各グループの (perGroup+1)〜(2*perGroup) 位 → バラージ（サバイバルトーナメント）
各グループの (2*perGroup+1) 位以降        → 敗退
```

2グループの場合 `perGroup = 6`。直接12人＋バラージ12人（4人がバラージを勝ち上がりTop16入り）＝ Top16。これはPDFの「Top16確定ライン」（合算順位1〜12位）「予選通過ライン」（13〜24位がサバイバルトーナメントへ）と一致する。

重要な点: この「グループごとの固定人数」方式は、**グループの人数が揃っている場合に限り**、2章の合算順位で1〜12位を取った場合と数学的に一致する（「グループ内順位が異なれば絶対優先」というルールにより、バケットKには各グループから必ず1人ずつしか入らないため）。同バケット内のタイブレーク方式（2.1）に関わらず、バケット単位の切り分けである4章の結論自体は変わらない。

### 4.2 3グループへの一般化

```
perGroup = 12 / 3 = 4
各グループ 1〜4位   → 直接勝ち上がり（計12人）
各グループ 5〜8位   → バラージ（計12人）
各グループ 9位以降  → 敗退
```

決勝進出枠の総数（Top16 = 直接12＋バラージ12）はグループ数によらず固定とする。

グループ人数が不揃い（例: 11/11/10人）でも、各グループが最低 `2 * perGroup = 8` 人いれば成立する（既存コードの `selectFinalsEntrantsByGroup` が持つバリデーションと同じ条件）。

### 4.3 スコープ外

決勝ブラケット内の**シード番号配置**（同グループ対決を避けるための並び順。2グループでは `TWO_GROUP_DIRECT_UPPER_SEEDS` / `TWO_GROUP_BARRAGE_SEED_TOKENS` という固定トークンマップで定義されている）は、本書の対象外とする。既存コードのコメント通り、別タスクとして切り出す。3グループの直接勝ち上がり・バラージの**対象者選定**（4.2）は本書で定義するが、**その並び順**は今後別途検討する。

**訂正**: 当初「CDM Excel自体は3グループ以上の決勝ブラケット構造をそもそも想定していない」と記載していたが、これは誤り。`cdm-export-design.md` 3.4の通り、決勝シートのブロック構成（Barrage1/Barrage2/Top16/UBQ/UBS/UBF/GF1/GF2、B3:B26のシードリスト）は固定レイアウトであり、予選のグループ数に関する情報を一切参照しない。`finals-group-selection.ts` の `selectFinalsEntrantsByGroup()` も、グループ数によらず常に同じ形（`directSeeds`: 16枠中12枠、`barrage`: 12枠）を返す契約になっている（`groupCount` はバリデーションにのみ使われ、出力の形は変わらない）。したがって**決勝ブラケットの構造自体は3グループ以上でも無改修で使え、シートの再設計は不要**。3グループ以上で未対応なのは、あくまで「その12+12枠に誰を割り当てるか」という**選定ロジック**（4.2の内容。現状 `selectFinalsEntrantsByGroup()` の3+グループ分岐は本書の合算順位ルールを使わず、単純なラウンドロビン式の仮並べのまま——コード内コメントに明記）と、上記の同グループ対決を避ける**並び順**（`TWO_GROUP_*_SEED_TOKENS`に相当する3グループ版が未設計）の2点のみである。

### 4.4 補足: 決勝進出の「パラメータ」と「シード決定方式」は別物

7章の質問4への回答で明確になった点として明記しておく。決勝進出ラインの**パラメータ**（`TOTAL_FINALS_SLOTS = 12`、直接/バラージ/敗退の人数区分）は、グループ数によらず共通であり、4.2はそれをそのまま3グループに当てはめたものに過ぎない。グループ数によって変わるのは、**どの選手がどのシード番号に入るかの決定方式**である:

- **2グループ・4グループ**（2.1の通りタイブレークが無くグループ文字順で確定）: シード枠と所属グループの対応は静的に決まる（例: 決勝の第1シードは常にグループAの1位）。
- **3グループ**（2.1の通りWDLスコア→得失点差でタイブレークする）: シード枠と所属グループの対応は各バケット内の実際の対戦成績に応じて動的に決まる（例: 決勝の第1シードは「バケット1」の最上位者であり、それがA/B/Cいずれのグループの1位かは大会結果次第）。

つまりこの「静的か動的か」の違いは2章（合算順位の決定ルール）がそのまま担っており、4章のパラメータ自体を3グループ用に別途作り直す必要はない。

## 6. 実装時の影響範囲（参考・本書では変更しない）

将来この仕様を実装する際に変更が想定される箇所:

- `src/lib/ranking-utils.ts` の `computeCombinedRanks()` — 現状はグループ内順位（バケット）を考慮しない素点・得失点差のみの簡易ソートで、本書のルールに未対応。7章質問1の決定により、実装目標は「バケット昇順→WDLスコア降順→得失点差降順」（シードは使わない）に確定
- `src/lib/finals-group-selection.ts` の `selectFinalsEntrantsByGroup()` — 3〜4グループが仮のラウンドロビン式並べのまま
- `src/components/tournament/group-setup-dialog.tsx` の `LOCKED_GROUP_COUNT = 2` — 3グループ以上のロック解除

## 7. 質問事項（大会運営確認済み・未決定を含む）

CDM Excelの実装（2章）が判明したことで生じた要検討事項。1・3・5は大会運営に確認し決定済み、2・4・6は引き続き未決定。

1. ~~**タイブレーク方式そのものの決定**: 現行CDM Excelの「同バケット内はグループ文字順で固定・タイブレークなし」を今後も正式仕様として踏襲するか、それとも初版が提案した「WDLスコア→シードランキング→得失点差→サドンデス」のようなタイブレークを新たに実装するか。~~
   **→ 決定**: シードランキングによるタイブレークは**導入しない**（初版の提案は不採用）。3グループ専用分岐と同じ「WDLスコア→得失点差」方式を、**グループ数によらず統一的に採用する**方針とする。これに伴い、現行の2グループ運用の「バケット内タイブレークなし（グループ文字順固定）」（2.1・3章）は、将来的に「WDLスコア→得失点差」方式に是正される想定。両方式で決着しない完全同点はサドンデスプレーオフで決着する（2.4参照）。
2. **添付PDFの誤り4件の真因**: 初版はPDFの誤りを「シードランキングでのタイブレーク漏れ」と解釈していたが、CDM Excelの数式にはシード参照が存在しない。誤りの実際の原因（手動修正・別バージョンの数式・入力ミス等）を大会運営側に確認できるか。3章の「A4/B4」「A15/B15」のような、CDM Excelの数式通りなら発生するはずの「Aが常に優先される」逆転が、実際にPDFの誤り4件と一致するかどうかも要突き合わせ。**未決定（引き続き要確認）。**
3. ~~**3グループ専用分岐の扱い**: `IF(グループ数=3, ...)` という特別分岐は他のグループ数（2, 4以上）に一般化すべきか。一般化する場合、シードランキングを組み込むかどうかも合わせて決める必要がある。~~
   **→ 決定**: 質問1と同じ決定に統合。シードランキングは組み込まず、「WDLスコア→得失点差」をグループ数によらず一般化する。
4. **4グループ超への対応**: `VSTACK`数式がA〜Dの4グループまでしかハードコードされていない。5グループ以上の大会を将来サポートする場合、CDM Excel側の数式改修が必要になるが、その優先度・要否を確認したい。**未決定（引き続き要確認）。**
5. ~~**不均等グループの実際の挙動**: 2.3の通り、CDM Excelが人数不揃いのグループ構成でどう振る舞うか（エラー／欠落／ズレ）は未検証。実データでの検証が必要か。~~
   **→ 決定**: 優先度低・検証不要。大会運営の運用上、グループごとの人数差は最大±1名に収まるよう調整されており、3グループ制自体が参加者合計37名以上の場合にのみ採用される。人数差が生じる場合の「最下位グループの余剰選手」は、いずれにせよ決勝進出ラインの対象外（4章のバラージ枠にも入らない下位バケット）になるため、Excel側の不均等時の挙動を厳密に検証する実務上の必要性は低いと判断する。
6. **アプリとCDM Excelの整合方針**: アプリ側で `computeCombinedRanks()` を本書のルールに沿って実装した場合、CDM Excelのエクスポート結果（`cdm-export`機能が出力するワークブック）はアプリの合算順位を数式として再現するのか、それとも従来通りCDM Excel自身の`SORTBY`数式に委ねて表示上の不一致を許容するのか。**未決定（引き続き要確認）。** ただし質問1の決定により、両者が目指すべき最終ルール（シードなし・WDLスコア→得失点差、グループ数によらず統一）自体は一致した。残る論点は、CDM Excel側の`SORTBY`数式（2グループ時に現在タイブレークが無い部分）を実際に改修するかどうかのみ。
