# 決勝進出者の選び方（2・3・4グループ）

対象: `selectFinalsEntrantsByGroup()`（`src/lib/finals-group-selection.ts`）が行う、予選グループから決勝進出者（直接勝ち上がり／バラージ／敗退）を選び、決勝ブラケットのシード番号（1〜24）を割り当てるロジック。

**現状**: UI・APIとも2グループ・3グループの両方をサポートしている（`group-setup-dialog.tsx` のグループ数セレクタ、`qualification-route.ts`、`finals-route.ts` の `TOP24_SUPPORTED_GROUP_COUNT`）。内部の `selectFinalsEntrantsByGroup()` は4グループも扱い、対応テストを維持しているが、4グループ以上をUI/APIから作成することは `docs/qualification-combined-ranking.md` §7の決定によりスコープ外である。

**改訂履歴**: 本書は当初「対象者選定はグループ数によらず統一するが、シード配置（同グループ対決回避）は2グループのみ手動トークンマップで固定・3グループ以上は一般アルゴリズム `assignAntiCollisionSeeds()` で対応」という設計を記載していた。しかしユーザー提供のCDM 2025公式結果ワークブック（実大会の確定結果）を検証したところ、実際のCDM 2025では同グループ対決の回避は一切行われておらず（1回戦に同グループ対戦が現に存在する）、シード配置は「対象者選定で決まった順位（バケット順→タイブレーク順）をそのまま連番でシード1〜24に採番するだけ」という単純な方式であることが判明した。これに伴い、2グループ専用トークンマップおよび `assignAntiCollisionSeeds()` は廃止し、以下の通りグループ数によらない単一ロジックに統一した。

## 1. 共通の枠組み

決勝進出枠の総数は常に固定値 `TOTAL_FINALS_SLOTS = 12`（参加人数から算出されるものではない）。

```
perGroup = 12 / グループ数

各グループの   1 〜 perGroup       位 → 直接勝ち上がり（Upper Bracket、計12人）
各グループの (perGroup+1) 〜 (2*perGroup) 位 → バラージ（サバイバルトーナメント、計12人）
各グループの (2*perGroup+1) 位以降              → 敗退
```

各グループには最低 `2 * perGroup` 人（直接進出分＋バラージ分）が必要（`selectFinalsEntrantsByGroup()` が持つバリデーション）。

「誰が直接勝ち上がり／バラージ対象になるか」（対象者選定）と「その中で決勝ブラケットのどのシード番号に入るか」（シード配置）は、以下の通り**同一のバケット結果からそのまま連番で決まる**。同グループ対決を避けるための特別な並び替えはグループ数によらず存在しない（CDM 2025の実大会でも同グループ対決は発生している）。

## 2. 対象者選定（バケット方式、2〜4グループ共通）

2グループでは各グループのグループ内順位がそのまま直接／バラージの判定になるため結果的にバケット方式と一致するが、3・4グループでは複数グループを1本の合算順位に横断させる必要がある。`docs/qualification-combined-ranking.md`（大会運営確認済みの決定）に基づき、グループ数によらず以下の「バケット」方式を採る:

1. 全グループの「1位どうし」を1つの束（バケット1）にする。「2位どうし」をバケット2に、以下同様。
2. バケットは絶対順位: **バケットNの全員は、バケットN+1の全員より必ず上位になる**（バケット内の個々の成績が劣っていても関係ない）。
3. 同じバケット内の並び順だけを、**WDLスコア（勝点）→得失点差**の順でタイブレークする。両方が完全同率なら、管理者が実施したグループ横断サドンデスの `combinedRankOverride` 順を使う。**シードランキングとグループ文字順は使わない**。

バケット1〜perGroup（直接進出）が計12人、バケット(perGroup+1)〜(2\*perGroup)（バラージ）が計12人になる。

## 3. シード配置（連番方式、2〜4グループ共通）

対象者選定で得られたバケット結果を、そのままの順序で連番のシード番号に採番する:

```
直接進出: バケット1〜perGroup の並び順のまま → シード1, 2, 3, ... 12
バラージ: バケット(perGroup+1)〜(2*perGroup) の並び順のまま → シード13, 14, 15, ... 24
```

`selectFinalsEntrantsByGroup()` は `{ directSeeds, barrageSeeds, groupCount }` を返し、`directSeeds[i].seed === i + 1`、`barrageSeeds[i].seed === TOTAL_FINALS_SLOTS + i + 1` が常に成り立つ（バケット方式で決まった並び順に対する単純な連番であり、成績以外の要素（グループ文字順・同グループ回避など）は一切考慮しない）。

決勝16人ブラケット側の対応（`generateBracketStructure(16)`・`generatePlayoffStructure(12)`、`src/lib/double-elimination.ts`）は、CDM 2025公式ブラケットの実データと一致するペアリングを持つ:

- 直接進出シード1〜12は、Upper Bracket 1回戦で `[[1,16],[8,9],[4,13],[5,12],[2,15],[7,10],[3,14],[6,11]]` の組み合わせに入る（13〜16は後述のバラージ勝者用の枠）。
- バラージ（シード13〜24）は12人によるプレーオフで、1回戦ペアが `[[17,24],[20,21],[18,23],[19,22]]`、シード16/13/15/14が不戦勝（BYE）でそのまま2回戦に進む。プレーオフの勝者は自身のシード番号（13〜16のいずれか）を保持したまま16人ブラケットに合流する（グループやバケットに応じた再割当ては行わない）。

### 3.1 具体例（2グループ）

グループA=14人、グループB=13人の場合、直接進出12人はバケット1〜6（各バケットにA・Bから1人ずつ、WDLスコア→得失点差でバケット内を並べ替え）の順にシード1〜12が振られる。グループ文字順に固定された配置（旧設計の `TWO_GROUP_DIRECT_UPPER_SEEDS` 等）ではなく、その大会の実際の成績次第でバケット内の順序（≒シードの主）が変わる点に注意。

### 3.2 具体例（3グループ）

3グループ・各9人以上で、バケット1（各グループの1位）の勝点が A1=5, B1=9, C1=7 だったとする。

```
バケット1の並び（勝点降順）: B1(9) → C1(7) → A1(5)
→ シード1=B1, シード2=C1, シード3=A1（バケット2以降も同様に続く）
```

グループ文字の並び順（A→B→C）ではなく、実際の勝点順になる。これは**その大会の実際の対戦結果によって決まる**（同じグループ構成でも大会ごとに並びが変わりうる）。3グループの場合、この並びが決勝ブラケットのシード番号にそのまま反映される（同グループ対決を避けるための組み替えは行わない）。

## 4. 4グループの場合（内部ロジック）

`perGroup = 3`。各グループの1〜3位が直接進出（計12人）、4〜6位がバラージ（計12人）、7位以降は敗退となる。選定・シード配置とも2・3グループと全く同じ「バケット方式→連番採番」ロジックを使う。

これは既存ロジックとテストの適用範囲であり、現在のUI/APIでは選択できない。

## 6. スコープ外・既知の制約

- **UI/APIからの4グループ以上**: `docs/qualification-combined-ranking.md` §7の決定によりスコープ外。4グループの内部ロジックは既存のまま残っているが、UI（`group-setup-dialog.tsx`）・API（`qualification-route.ts`、`finals-route.ts`）のいずれからも選択・作成できない。5グループ以上は内部ロジックも対象外である。
- **CDM Excelテンプレート側の数式**: `cdm-2025-template.xlsm` の `SORTBY` 数式は今回変更していない（同§7 質問6の決定）。アプリ側の合算順位・決勝進出ロジックとCDM Excel側の表示が一致しない期間が生じうる。
