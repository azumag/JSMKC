# 決勝進出者の選び方（2グループ／3グループ）

対象: `selectFinalsEntrantsByGroup()`（`src/lib/finals-group-selection.ts`）が行う、予選グループから決勝進出者（直接勝ち上がり／バラージ／敗退）を選ぶロジック。

**現状**: UI・APIではグループ数はまだ2に固定されています（`group-setup-dialog.tsx` の `LOCKED_GROUP_COUNT = 2`）。本書の「3グループの場合」はコード上は実装・テスト済みですが、UIから3グループのトーナメントを作成することはまだできません（別途のUI/API解禁作業が必要。詳細は5章）。

## 1. 共通の枠組み

決勝進出枠の総数は常に固定値 `TOTAL_FINALS_SLOTS = 12`（参加人数から算出されるものではない）。

```
perGroup = 12 / グループ数

各グループの   1 〜 perGroup       位 → 直接勝ち上がり（Upper Bracket、計12人）
各グループの (perGroup+1) 〜 (2*perGroup) 位 → バラージ（サバイバルトーナメント、計12人）
各グループの (2*perGroup+1) 位以降              → 敗退
```

各グループには最低 `2 * perGroup` 人（直接進出分＋バラージ分）が必要（`selectFinalsEntrantsByGroup()` が持つバリデーション）。

「誰が直接勝ち上がり／バラージ対象になるか」（本書の対象）と、「その中で決勝ブラケットのどのシード位置に入るか」（同グループ対決を避けるための並び順）は別の問題である点に注意。後者は現状2グループでのみ設計されている（2.2節）。

## 2. 2グループの場合（現行本番運用）

`perGroup = 6`。直接12人＋バラージ12人（4人がバラージを勝ち上がりTop16入り）で決勝Top16を構成する。

### 2.1 選定方法

2グループの場合、「誰が直接／バラージ対象か」は各グループの**グループ内順位そのもの**で決まる。2グループ間で成績を比較する必要はない（グループAの1〜6位、グループBの1〜6位が機械的に直接進出、7〜12位がバラージ、というだけ）。

### 2.2 シード配置（同グループ対決回避）

2グループの場合はさらに、「直接進出した12人が決勝ブラケットのどのシード番号に入るか」まで固定のトークンマップで決まっている（`TWO_GROUP_DIRECT_UPPER_SEEDS` / `TWO_GROUP_BARRAGE_SEED_TOKENS`）。これは序盤の対戦で同グループ同士がぶつかるのを避けるために手動で設計された配置である。

例（グループA=14人、グループB=13人の場合）:

```
直接勝ち上がり（シード → 選手）:
  1:A1  2:B3  3:B1  4:A3  5:B2  6:A4
  7:A2  8:B4  9:A5  11:B5  13:B6  15:A6

バラージ（プレーオフシード1〜12）:
  B8, B7, A8, A7, B9, A11, B10, A12, A10, B12, A9, B11
```

16人ブラケットのUpper R1（1回戦）は上から順に次のようになる（A1 vs バラージ、B4 vs A5、…）:

```
A1 vs バラージ    B4 vs A5
B2 vs バラージ    A3 vs B6
B1 vs バラージ    A4 vs B5
A2 vs バラージ    B3 vs A6
```

このシード配置は成績（勝点・得失点差など）を一切見ない、純粋にグループ内順位の位置だけで決まる固定パターンである。

## 3. 3グループの場合（実装済み・UI未解禁）

`perGroup = 4`。各グループの1〜4位が直接進出（計12人）、5〜8位がバラージ（計12人）、9位以降は敗退。

### 3.1 選定方法（バケット方式）

3グループでは「誰が直接／バラージ対象か」を単純にグループごとに固定で決めることができない（3グループを1本の合算順位に横断的に並べる必要がある）。`docs/qualification-combined-ranking.md`（大会運営確認済みの決定）に基づき、以下の「バケット」方式を採る:

1. 全グループの「1位どうし」を1つの束（バケット1）にする。「2位どうし」をバケット2に、以下同様。
2. バケットは絶対順位: **バケットNの全員は、バケットN+1の全員より必ず上位になる**（バケット内の個々の成績が劣っていても関係ない）。
3. 同じバケット内の並び順だけを、**WDLスコア（勝点）→得失点差**の順でタイブレークする。**シードランキングは一切使わない**（大会運営の決定により不採用）。

バケット1〜4（`perGroup=4`なので）が直接進出、バケット5〜8がバラージとなる。

### 3.2 具体例

3グループ・各9人以上（`__tests__/lib/finals-group-selection.test.ts` のテストケースと同じ数値）で、バケット1（各グループの1位）の勝点が A1=5, B1=9, C1=7 だったとする。

```
バケット1の並び（勝点降順）: B1(9) → C1(7) → A1(5)
```

グループ文字の並び順（A→B→C）ではなく、実際の勝点順になる。これは2グループの2.2節のような固定トークンマップとは異なり、**その大会の実際の対戦結果によって決まる**（同じグループ構成でも大会ごとに並びが変わりうる）。

### 3.3 シード配置は未設計（既知の制約）

3グループの場合、直接進出・バラージの**対象者選定**（3.1節）は実装済みだが、決勝ブラケットの**どのシード位置に入るか**（2.2節の同グループ対決回避に相当するもの）はまだ設計されていない。現状はバケット順（1バケット目の中の勝点順、2バケット目の中の勝点順、…）にそのまま1番から順にシード番号を振っているだけで、同グループ対決を避ける工夫は入っていない。これは別タスクとして残されている。

## 4. 比較表

| | 2グループ（現行） | 3グループ（実装済み・未解禁） |
|---|---|---|
| perGroup | 6 | 4 |
| 直接進出の選定 | 各グループの1〜6位（グループ内順位そのまま） | バケット1〜4（各グループのk位を束ねて勝点→得失点差で並べる） |
| バラージの選定 | 各グループの7〜12位 | バケット5〜8 |
| グループ間の成績比較 | 不要 | 必要（同バケット内でWDLスコア→得失点差により比較） |
| シード配置（同グループ対決回避） | 手動設計の固定トークンマップあり | 未設計（バケット順にそのまま採番） |
| UIから利用可能か | 可能（現行本番運用） | 不可（`LOCKED_GROUP_COUNT=2` により作成不可） |
