# 決勝進出者の選び方（2・3・4グループ）

対象: `selectFinalsEntrantsByGroup()`（`src/lib/finals-group-selection.ts`）が行う、予選グループから決勝進出者（直接勝ち上がり／バラージ／敗退）を選ぶロジック。

**現状**: UI・APIとも2グループ・3グループの両方をサポートしている（`group-setup-dialog.tsx` のグループ数セレクタ、`qualification-route.ts`、`finals-route.ts` の `TOP24_SUPPORTED_GROUP_COUNT`）。内部の `selectFinalsEntrantsByGroup()` は4グループも扱い、対応テストを維持しているが、4グループ以上をUI/APIから作成することは `docs/qualification-combined-ranking.md` §7の決定によりスコープ外である。

## 1. 共通の枠組み

決勝進出枠の総数は常に固定値 `TOTAL_FINALS_SLOTS = 12`（参加人数から算出されるものではない）。

```
perGroup = 12 / グループ数

各グループの   1 〜 perGroup       位 → 直接勝ち上がり（Upper Bracket、計12人）
各グループの (perGroup+1) 〜 (2*perGroup) 位 → バラージ（サバイバルトーナメント、計12人）
各グループの (2*perGroup+1) 位以降              → 敗退
```

各グループには最低 `2 * perGroup` 人（直接進出分＋バラージ分）が必要（`selectFinalsEntrantsByGroup()` が持つバリデーション）。

「誰が直接勝ち上がり／バラージ対象になるか」（対象者選定）と、「その中で決勝ブラケットのどのシード位置に入るか」（同グループ対決を避けるための並び順、シード配置）は別の問題である点に注意。2グループは固定トークンマップ（2.2節）、3グループは一般アルゴリズム `assignAntiCollisionSeeds()`（3.3節）で、どちらも実装済み。

## 2. 2グループの場合（現行本番運用）

`perGroup = 6`。直接12人＋バラージ12人（4人がバラージを勝ち上がりTop16入り）で決勝Top16を構成する。

### 2.1 選定方法

2グループの場合、「誰が直接／バラージ対象か」は各グループの**グループ内順位そのもの**で決まる。2グループ間で成績を比較する必要はない（グループAの1〜6位、グループBの1〜6位が機械的に直接進出、7〜12位がバラージ、というだけ）。

### 2.2 シード配置（同グループ対決回避）

2グループの場合はさらに、「直接進出した12人が決勝ブラケットのどのシード番号に入るか」まで固定のトークンマップで決まっている（`TWO_GROUP_DIRECT_UPPER_SEEDS` / `TWO_GROUP_BARRAGE_SEED_TOKENS`）。これは序盤の対戦で同グループ同士がぶつかるのを避けるために手動で設計された配置である。

例（グループA=14人、グループB=13人の場合）:

```
直接勝ち上がり（シード → 選手）:
  1:A1  2:B3  3:B1  4:A3  5:B2  6:A4
  7:A2  8:B4  9:A5  11:B5  13:B6  15:A6

バラージ（プレーオフシード1〜12）:
  B8, B7, A8, A7, B9, A11, B10, A12, A10, B12, A9, B11
```

16人ブラケットのUpper R1（1回戦）は上から順に次のようになる（A1 vs バラージ、B4 vs A5、…）:

```
A1 vs バラージ    B4 vs A5
B2 vs バラージ    A3 vs B6
B1 vs バラージ    A4 vs B5
A2 vs バラージ    B3 vs A6
```

このシード配置は成績（勝点・得失点差など）を一切見ない、純粋にグループ内順位の位置だけで決まる固定パターンである。

## 3. 3グループの場合

`perGroup = 4`。各グループの1〜4位が直接進出（計12人）、5〜8位がバラージ（計12人）、9位以降は敗退。

### 3.1 選定方法（バケット方式）

3グループでは「誰が直接／バラージ対象か」を単純にグループごとに固定で決めることができない（3グループを1本の合算順位に横断的に並べる必要がある）。`docs/qualification-combined-ranking.md`（大会運営確認済みの決定）に基づき、以下の「バケット」方式を採る:

1. 全グループの「1位どうし」を1つの束（バケット1）にする。「2位どうし」をバケット2に、以下同様。
2. バケットは絶対順位: **バケットNの全員は、バケットN+1の全員より必ず上位になる**（バケット内の個々の成績が劣っていても関係ない）。
3. 同じバケット内の並び順だけを、**WDLスコア（勝点）→得失点差**の順でタイブレークする。両方が完全同率なら、管理者が実施したグループ横断サドンデスの `combinedRankOverride` 順を使う。**シードランキングとグループ文字順は使わない**。

バケット1〜4（`perGroup=4`なので）が直接進出、バケット5〜8がバラージとなる。

### 3.2 具体例

3グループ・各9人以上（`__tests__/lib/finals-group-selection.test.ts` のテストケースと同じ数値）で、バケット1（各グループの1位）の勝点が A1=5, B1=9, C1=7 だったとする。

```
バケット1の並び（勝点降順）: B1(9) → C1(7) → A1(5)
```

グループ文字の並び順（A→B→C）ではなく、実際の勝点順になる。これは2グループの2.2節のような固定トークンマップとは異なり、**その大会の実際の対戦結果によって決まる**（同じグループ構成でも大会ごとに並びが変わりうる）。

### 3.3 シード配置（同グループ対決回避、一般アルゴリズム）

3グループの場合、決勝ブラケットの**どのシード位置に入るか**（2.2節の同グループ対決回避に相当するもの）は、2グループのような固定トークンマップでは実現できない（3.2節の通りバケットの中身が大会結果に応じて変わるため、「Aのn位はこのシード」という固定表は組めない）。代わりに `assignAntiCollisionSeeds()`（`finals-group-selection.ts`）という一般アルゴリズムで実装している:

1. 決勝ブラケット（16人）・バラージ（12人）それぞれについて、「1回戦で対戦相手が決まっているシード（＝同時点でグループが分かる2つの直接進出シードの組）」と「1回戦の相手がまだ決まっていないシード（＝バラージ勝者待ち）」を、実際のブラケット生成ロジック（`generateBracketStructure`/`generatePlayoffStructure`）から機械的に導出する（ハードコードされた組ではなく、ブラケット構造そのものから毎回導出するため、ブラケット側の実装が変わってもズレない）。
2. 「相手未定」のシード（決勝: 1・3・5・7位、バラージ: 1〜4位のBYE枠）には、全体順位（バケット順→タイブレーク順）の上位から順に割り当てる。
3. 「1回戦の相手が決まっている」組には、全体順位の残り上位から2人ずつ取り出し、**異なるグループの組み合わせになるまで**次点の選手と組ませる、という操作を順位の高い方から機械的に繰り返す。
4. バラージ終了後のPhase 2では4人の勝者グループが確定しているため、`reseedDirectEntrantsAgainstPlayoffWinners()` が直接進出12人を同じシード集合内で再配置する。これにより、直接進出者同士だけでなく「直接進出者 vs バラージ勝者」を含むUpper R1の全8試合で同グループ対決を避ける。進出者の顔ぶれ自体は変えない。

3グループ・各グループ最低4人（直接進出分）という構成では、どのバケットも必ず3グループすべてから1人ずつ含まれるため、この組み合わせ操作は必ず異なるグループの組を作れる（数学的に破綻しない）。

### 3.4 具体例（決勝シード配置）

3グループ・各9人、全員の勝点が横並び（タイ）の場合、`assignAntiCollisionSeeds()` は次のように配置する:

```
直接進出（シード → 選手）:
  1:A1  2:B2  3:B1  4:A3  5:C1  6:C3
  7:A2  8:B4  9:C4  11:A4  13:B3  15:C2
```

1回戦で対戦が決まっている組（シード2/15, 4/13, 6/11, 8/9）はすべて異なるグループどうしになる（B2/C2、A3/B3、C3/A4、B4/C4）。

## 4. 4グループの場合（内部ロジック）

`perGroup = 3`。各グループの1〜3位が直接進出（計12人）、4〜6位がバラージ（計12人）、7位以降は敗退となる。選定は3グループと同じバケット方式で、同じグループ内順位の選手を束ね、WDLスコア→得失点差で並べる。`assignAntiCollisionSeeds()` も同じ一般アルゴリズムで、既知の1回戦ペアを異なるグループに配置する。

これは既存ロジックとテストの適用範囲であり、現在のUI/APIでは選択できない。

## 5. 比較表

| | 2グループ（現行） | 3グループ | 4グループ（内部ロジック） |
|---|---|---|
| perGroup | 6 | 4 | 3 |
| 直接進出の選定 | 各グループの1〜6位（グループ内順位そのまま） | バケット1〜4（各グループのk位を束ねて勝点→得失点差で並べる） | バケット1〜3 |
| バラージの選定 | 各グループの7〜12位 | バケット5〜8 | バケット4〜6 |
| グループ間の成績比較 | 不要 | 必要（同バケット内でWDLスコア→得失点差により比較） | 必要（同左） |
| シード配置（同グループ対決回避） | 手動設計の固定トークンマップ | 一般アルゴリズム（`assignAntiCollisionSeeds()`） | 一般アルゴリズム（同左） |
| UIから利用可能か | 可能 | 可能 | 不可 |

## 6. スコープ外・既知の制約

- **UI/APIからの4グループ以上**: `docs/qualification-combined-ranking.md` §7の決定によりスコープ外。4グループの内部ロジックは既存のまま残っているが、UI（`group-setup-dialog.tsx`）・API（`qualification-route.ts`、`finals-route.ts`）のいずれからも選択・作成できない。5グループ以上は内部ロジックも対象外である。
- **CDM Excelテンプレート側の数式**: `cdm-2025-template.xlsm` の `SORTBY` 数式は今回変更していない（同§7 質問6の決定）。アプリ側の合算順位・決勝進出ロジックとCDM Excel側の表示が一致しない期間が生じうる。
